Compromised vs Locked-Out: The 4-Step Order

A locked-out user has lost access to their own account. A compromised user has lost it to someone else. Conflating the two is how attackers keep their foothold while you "fix" the problem.

Why a password reset alone is not enough

If a user simply forgot their password, an admin reset solves it. But for a compromised account, the password is just one of several footholds. Attackers establish persistence through mechanisms that a password change does not touch:

  • Active sessions and sign-in cookies
  • OAuth grants to third-party apps
  • App passwords
  • Mail forwarding rules and filters
  • Delegated mailbox access and Drive sharing

Reset the password without addressing these, and the attacker keeps reading mail, exfiltrating Drive, and re-entering at will.

Google's official 4-step order

  1. SUSPEND the account first. Per Google, "Suspending a user resets the user's sign-in cookies and OAuth tokens." This immediately contains the attacker by killing active sessions before you investigate.
  2. INVESTIGATE. For a compromised admin, review the Admin audit logs for recent configuration changes first. Then review devices (wipe suspicious ones), check User log events for sign-in activity, run an email log search, and review security reports for OAuth, Drive, Calendar, and Groups exposure.
  3. REVOKE access. Reset the password, revoke OAuth 2.0 tokens, and remove any app passwords. This tears down the persistence mechanisms the investigation surfaced.
  4. RESTORE. Unsuspend the account, issue a temporary password (the user sets a new, unique one), enroll 2SV / security keys, and run the Gmail security checklist — filters, forwarding, delegation, and recovery options.
Resetting the password is step 3, not step 1Suspend first to kill active sessions and tokens, or the attacker keeps their foothold while you work. A password change on a live session does nothing to evict an attacker who is already inside.

Two gaps in automatic token revocation

Don't assume revocation is total. Two known caveats:

  • Token revocation does not cover Apps Script-based apps — review and remove these separately.
  • A password change initiated from an Android device will not revoke that device's sync token, leaving a path open from the very device the attacker may control.
Verify in your consoleConfirm the current location and labels for suspend, OAuth token revocation, app passwords, and the security investigation tools before an incident — console paths drift, and you don't want to be hunting for the suspend button while an attacker is active. Walk the full 4-step order once on a test account so the muscle memory is there when it counts.
Director takeawaySuspend, investigate, revoke, restore — in that order. The single most important sequencing decision is suspending before resetting, because suspension is what kills the live cookies and tokens. Treat Apps Script apps and Android-initiated changes as known revocation blind spots and clear them by hand.
