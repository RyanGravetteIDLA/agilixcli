Module 3 · Admin-Assisted User Recovery

Suspend / Reactivate & the Self-Recovery Toggle

Reactivating a suspended account is a two-click job — unless Google suspended it. And the one self-service password toggle you'll be asked about is largely off-limits for K-12 students.

Reactivate a suspended user

  1. Go to Directory → Users and filter User status = Suspended.
  2. On the user's row, click More options → Reactivate.
  3. Confirm.

For many accounts at once, reactivate in bulk via CSV upload (set New Status = Active) or through the Admin SDK Directory API.

Hard constraint“You can't restore an account that was suspended for abuse or for breaching the Google Workspace or Cloud Identity Terms of Service.” Those Google-initiated suspensions can only be lifted by Google Support — the Reactivate button won't help.

The user self-recovery toggle

The setting “Allow users and non-super admins to recover their account” lives at Security → Authentication → Account recovery.

  • When ON — non-admin users can reset their own password using a recovery phone or email they previously configured. Note: users with 2SV enabled can only use a recovery email, not a phone.
  • When OFF — users must contact an admin to recover access.

This toggle is incompatible with third-party SSO and Password Sync — in those environments account recovery happens in your IdP, not Google.

K-12 criticalThe self-recovery setting is “unavailable for Google Workspace for Education users under the age of 18.” Super admins can enable it for higher-education accounts and other admin users, but for most K-12 student populations self-recovery is simply not available. Student password resets route through the admin — back to L3.1. This is the single most important point of this module for a K-12 director.

What is NOT a recovery action

Changing a user's name or moving them to a different OU does not delete the account and needs no “recovery.” These are routine edits — don't confuse them with deletion or suspension workflows.

Verify in your consoleConfirm the exact setting name and location under Security → Authentication → Account recovery, and the Reactivate action on a suspended user's row — Google's labels and the under-18 availability rules drift. Director takeawayReactivation is easy for admin-initiated suspensions and impossible (without Google Support) for abuse/ToS suspensions. And don't build a student-facing self-service password plan around the recovery toggle — it's blocked for under-18 Education users, so your students' resets stay an admin responsibility.
