Module 3 · Admin-Assisted User Recovery

2SV Lockouts: Backup Codes, Lost Keys & Passkeys

When a user loses their phone or security key, you can't re-enroll them — but you can hand them a backup code and clear lost factors so they can re-enroll themselves.

The key principle

Per Google: “Only the user can turn on 2-Step Verification.” An admin cannot enroll 2SV on a user's behalf. Your primary lockout remedy is to provide a backup code that lets the user sign in once and then fix their own factors.

Get backup codes for a locked-out user

  1. Go to Directory → Users → [user] → Security → 2-Step Verification → Get backup verification codes.
  2. Copy an existing unused code and read it to the user, or click Generate new codes.

“Creating new verification codes invalidates any existing code.” Generate new codes only if you suspect the existing codes were stolen or used up — otherwise just copy an existing one.

Admin-vs-admin ruleOnly super administrators can generate backup codes for another admin account. And users who are required to use a security key cannot self-generate backup codes — an admin must provide them.

Turning off 2SV (rarely)

Turning off 2SV for a user is possible but “isn't recommended.” It is done by moving the user to an OU or group where 2SV enforcement doesn't apply — there is no per-user toggle.

Don't conflate two features“Login challenges” (extra verification prompts at sign-in) is a separate feature from “2SV enforcement.” Resetting login challenges does not change a user's 2SV status, and vice versa.

Lost security key or passkey

You remove the lost factor; the user re-enrolls a new one. Two-part job:

Admin side — remove the lost factor

  1. Go to Directory → Users → [user] → Security → Passkeys & security keys.
  2. Hover the lost key's row and click Remove.

Google's guidance: “Remove a security key only when the key is lost.”

User side — re-enroll

The user then enrolls their own new key or passkey from their Google Account → Security → 2-Step Verification. The “Move to a new phone” flow is a user-side re-enrollment workflow, not an admin feature — point the user to it; you can't do it for them.

Sign-in cookies vs. 2SV

The separate Reset sign-in cookies security action signs the user out of all sessions everywhere. It does not delete any 2SV factors — use it for a suspected session hijack, not as a 2SV fix.

Verify in your consoleConfirm the labels Get backup verification codes, Passkeys & security keys, and Reset sign-in cookies under the user's Security tab — Google renames these periodically. The security-settings article now lives at knowledge.workspace.google.com/admin/… (old support.google.com/a/… links redirect).
Director takeawayYou are the backup-code provider, not the 2SV enroller. For a phone/key loss: hand the user a backup code (super admin required for admin accounts), remove the lost factor from Security → Passkeys & security keys, and have the user re-enroll their own new factor. Keep “2SV enforcement,” “login challenges,” and “sign-in cookies” mentally separate.
