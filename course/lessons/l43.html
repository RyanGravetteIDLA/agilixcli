Passkeys, Passwordless & Advanced Protection

Passkeys do two different jobs, and conflating them causes misconfiguration. Layer the Advanced Protection Program on top for your highest-risk staff — but only after you understand what it does to recovery.

Passkeys play two distinct roles

The same credential serves two purposes, configured in two different places:

  • As a 2SV second factor — governed under your 2SV settings via the "Only security key" option (which, as covered in L4.1, accepts passkeys too).
  • To skip the password entirely — passwordless sign-in, governed by a separate setting below.

These are independent. Enabling one does not enable the other.

The skip-passwords setting

The exact control is "Allow users to skip their password and authenticate with a passkey," at Security → Authentication → Passwordless → Skip passwords. It is OFF by default. Key behaviors:

  • When off, users cannot skip the password — but they can still create and use passkeys as a 2SV method.
  • Enabling skip-passwords does not remove the user's password; the password fallback remains.
  • Enrolling a passkey does not auto-enable 2SV.
  • Disabling skip-passwords later still retains the user's passkeys for 2SV use.

Passkey loss and recovery

There is no passkey-specific recovery path. A lost passkey falls back to the general 2SV recovery framework:

  • Backup codes (user-generated or admin-generated — see L4.4).
  • Admin removal of the lost passkey from the account.
  • Enrolling a second passkey or security key in advance as a spare.

Because the password remains as a fallback, passwordless is lower-risk than it sounds — but the moment you also enforce phishing-resistant-only 2SV, that safety margin disappears and backup codes become essential.

Advanced Protection Program (APP)

APP is Google's bundle of strongest, non-overridable policies for high-risk users: super and delegated admins, leadership, finance, and IT. Configure it at Security → Authentication → Advanced Protection Program. Enrolled users get:

  • Sign-in via security keys or passkeys only.
  • High-risk third-party app scopes blocked unless you, the admin, have trusted them.
  • Enhanced scanning of incoming threats.

As of July 2024, passkeys became a valid APP enrollment option — previously enrollment required two physical security keys.

APP hardens recovery on purposeAPP intentionally makes account recovery harder. If an enrolled user loses all their factors, recovery triggers a multi-day Google identity-verification delay ("a few days"). There is no admin shortcut around it. Mitigation: APP enrollment requires a backup method — either a recovery email + phone, or an additional passkey / security key. Pre-stage backup factors BEFORE you enroll any high-risk staff member, or a single lost key can sideline an administrator for days.
Verify in your consoleConfirm the skip-passwords default is still Off in your tenant at Security → Authentication → Passwordless, and review the current APP enrollment requirements at Security → Authentication → Advanced Protection Program before onboarding staff. Google's labels and defaults in the passwordless/APP area shift frequently.
Director takeawayKeep the two passkey roles straight: 2SV factor vs. passwordless sign-in. Reserve APP for genuinely high-risk accounts, and never enroll anyone in APP — or remove their password fallback — without a second factor and backup codes already staged. The stronger the control, the more deliberate the recovery prep.
