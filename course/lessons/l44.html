The Backup-Codes Program

Backup codes are the linchpin of every control in this module. Each hardening step you take — enforcement, passkeys, APP — raises lockout risk, and backup codes are the fallback that turns a lockout into a self-service recovery.

How backup codes work

  • A set is ten single-use codes. Each code works exactly once.
  • Generating a new set invalidates the old set.
  • Admin-generated and user-generated sets invalidate each other — only the most recently generated set is valid.

User self-generation

Users can mint their own set from their personal account security page:

Google Account → Security → 2-Step Verification → Backup codes — then print or download.

Admin generation

When a user can't self-serve, an admin generates codes for them:

Directory → Users → [user] → Security → 2-Step Verification → Get backup verification codes

  • Requires the User management privilege.
  • Only super administrators can generate backup codes for another admin account.
  • Users required to use a security key cannot self-generate — an admin must provide their codes.

Storage best practice

Treat backup codes as sensitive credentials:

  • Print or download them — store offline, "like a passport."
  • Never send them in plaintext email or paste them into chat/tickets.
  • Regenerate after use so a partially-spent set doesn't linger.

Why this is the recovery linchpin

Backup codes are the primary fallback recovery path when a phone, passkey, or security key is lost — the mechanism that prevents enforcement-day lockouts. They tie the whole module together:

  • L4.2 (enforcement): enforce 2SV ↔ pre-stage backup codes so no one is stranded at cutover.
  • L4.3 (APP): APP's multi-day recovery delay ↔ backup codes and pre-staged factors are your only fast path around it.
Lockout warningFor break-glass super admin accounts, a lost-or-no second factor with no valid backup codes can mean a multi-day Google recovery — or, under APP, no admin override at all. Generate and securely store a current set for every super admin before you enforce 2SV, and regenerate any set after it's been used or its secrecy is in doubt.
Verify in your consoleConfirm the admin generation path at Directory → Users → [user] → Security → 2-Step Verification — Google occasionally relabels the "Get backup verification codes" action and the privilege names. Verify your User management privilege scoping before relying on a help-desk admin to issue codes.
Director takeawayBake "generate + securely distribute backup codes" into both your 2SV rollout and your break-glass admin setup. Codes are cheap insurance: a current, offline set per super admin and a documented help-desk procedure for staff turns the lockouts this module's controls create into routine, self-service recoveries.
