Audit & Monitoring

When an account is compromised, the logs you didn't enable last quarter are the ones you'll wish you had. Audit visibility is a standing posture, not an incident-time scramble.

Your central console: Audit and investigation

Almost everything starts at Reporting → Audit and investigation. This is the unified surface for querying log events across services. Two log sources carry most of your incident-response weight:

Admin log events

Captures configuration changes: an admin being added, a role or privilege being granted, a service turned on, recovery settings being changed. When you suspect a compromised admin, the Admin audit log is where you reconstruct what the attacker touched — review it for recent configuration changes first.

User log events / Login audit

Captures sign-in activity: successful and failed sign-ins, suspicious logins, password changes, and 2-Step Verification (2SV) events. This is where you confirm whether an account was actually accessed and from where.

Alert center: let the system watch for you

The Alert center at Security → Alert center surfaces system-defined rules that fire automatically. Keep these enabled and reviewed:

  • Suspicious login
  • Government-backed attacks
  • User granted Admin privilege
  • User suspended (by an admin, and due to suspicious activity)
  • Leaked-password and phishing alerts

Manage these rules at Reporting → Rules → system-defined rules, and configure email notifications so the right people are paged without having to live in the console.

Recovery-specific events to watch

For this module's threat model — account-recovery abuse and admin self-escalation — these are the highest-signal events:

Event to watchWhere to find itWhy it matters
Change to "Super admin account recovery" or "Recovery information" settingsAdmin log eventsAn attacker loosening recovery controls to create a durable re-entry path
New recovery email or phone added to an accountAdmin log events / User log eventsClassic account-takeover persistence — attacker seeds their own recovery channel
User granted Admin privilegeAdmin log events + Alert centerPrivilege escalation; an attacker turning a foothold into tenant-wide control
Suspicious login / failed sign-in spikesUser log events / Login auditActive brute-force or credential-stuffing against an account
Password change followed by 2SV enrollment changeUser log eventsAttacker locking the legitimate owner out after takeover
Verify in your consoleAn April 2026 Workspace update expanded audit-log functionality and added event fields in the Admin console. Field names, available filters, and even the exact location of some events may differ from what you remember — re-confirm your saved investigations and alert rules at Reporting → Audit and investigation and Reporting → Rules against the current UI before relying on them in an incident.
Director takeawayStand up monitoring before you need it. Keep the system-defined alert rules enabled with email notifications wired to your team, and know cold which log answers which question: Admin log events for "what was changed," User log events for "who signed in." Recovery-setting changes and "User granted Admin privilege" are your canary events for account-recovery abuse.
