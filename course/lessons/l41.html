2SV Methods in 2026

Choosing 2-Step Verification methods is the first hardening decision in your security stack. Each method trades convenience for phishing resistance, and each carries a distinct lockout and recovery profile.

Where to configure it

2SV policy lives at Security → Authentication → 2-Step Verification and requires a super admin. Settings apply per organizational unit (OU) or configuration group, so you can pilot before going district-wide.

Supported methods

As of 2026, Google Workspace supports the following second factors:

  • Security keys — FIDO hardware keys, including Google's Titan keys.
  • Passkeys — FIDO credentials bound to a device or synced via a provider.
  • Google prompt — a "Yes/No" push to a signed-in device.
  • Google Authenticator (TOTP) — time-based one-time codes from an app.
  • Backup codes — ten single-use codes (covered in L4.4).
  • SMS / voice codes — codes delivered by text or phone call.

The "Allowed 2-Step Verification methods" dropdown

This dropdown is the lever that determines which factors your users may rely on. Three options matter:

OptionWhat it does
AnyPermits all supported methods. Most permissive; lowest lockout risk.
Any except verification codes via text, phone callRemoves SMS/voice. Users who currently verify via text or call will be locked out until they enroll a permitted method.
Only security keyRestricts to phishing-resistant factors. Per Google, this option now accepts both security keys and passkeys — they sit at the same phishing-resistance level.

What to recommend, what to retire

Strongest: security keys and passkeys. Both are FIDO-based and phishing-resistant — the credential cannot be replayed against a spoofed sign-in page.

Discouraged: SMS. In Google's words, "Text messages are discouraged as they rely on external carrier networks and might be intercepted." The direction of travel is clear: SMS out, passkeys in. Treat SMS as a transitional fallback, not a destination.

Phishing resistance at a glance

MethodPhishing resistanceRecovery note
Security key / passkeyHighEnroll a second key in advance; otherwise falls back to backup codes / admin action.
Google Authenticator (TOTP)MediumRe-enrollment needed if the device is lost; backup codes bridge the gap.
Google promptMediumTied to a signed-in device; backup codes bridge the gap.
SMS / voiceLowCarrier-dependent; interceptable. Avoid as a primary factor.

The 2024 enrollment change you should know

In May 2024 Google simplified 2SV enrollment, and the behavior persists in 2026:

  • Users add Authenticator or a security key before turning on 2SV — a phone is no longer a prerequisite.
  • A user who self-disables 2SV now keeps their enrolled methods, so re-enabling is fast.
  • When an admin removes a user's 2SV, enrolled methods are still cleared — that user must re-enroll from scratch.
Verify in your consoleGoogle relabels and re-defaults these options periodically. Confirm the exact dropdown wording at Security → Authentication → 2-Step Verification in your own console, and check whether "Only security key" in your tenant lists passkeys explicitly before you rely on it.
Director takeawayStandardize on security keys and passkeys for staff, keep Authenticator as a practical middle tier, and phase SMS out. Every method you restrict raises lockout risk — never enable a stricter "Allowed methods" setting without first confirming users hold a permitted factor and have backup codes in hand.
